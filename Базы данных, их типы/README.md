### Задание 1. СУБД

### Кейс
Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать 
правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для
каждой предметной области. 

Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему? 
 
1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков.
СУБД должна гарантировать целостность и чёткую структуру данных.

1.1.* Хеширование стало занимать длительно время, какое API можно использовать для ускорения работы? 

1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к 
маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? 
СУБД должны быть гибкими и быстрыми.

1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?

1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это 
реализовать?

1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов 
по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать

1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД

1.5.* Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?

### Ответ 1
1.1. Четкая структура и целостность - какая-нибудь реляционную СУБД. Например PostgreSQL
1.2. NoSQL (Например MongoDB) для лендингов, т.к. у каждого может быть своя структура и NoSQL подойдет хорошо. Для CRM наоборот, т.к. структура одна, логичнее использовать SQL-СУБД (тот же PostgreSQL)
1.2.* В теории можно, если страницы лендингов маппить в какой-нибудь формат и запихивать в таблицу. Но, как по мне, логичнее использовать MongoDB
1.3. Опять же, простая и понятная структура = SQL. PostgreSQL/MySQL
1.3.* Можно, наверное, использовать БД для CRM, создав отдельные таблицы и связи, если будут нужны. Если не будет никаких связей, но будто и нет смысла хранить все в одной БД
1.4. Логистика, быстрый поиск по связам = какая-нибудь графовая БД
1.4.* Можно, если в 1.4 использовать реляционную. Закупки и логистика звучат как тесно связанные вещи, так что может в этом и есть смысл
1.5. Можно, если извратиться и нагородить костылей. Но я бы так не делал

### Ответ 4.

После 30 минут рассказов про теорему CAP я должен выбрать какую СУБД использовать, учитывая, что я без понятия что там будет храниться?
Я без понятия какие есть моделы распределенных вычислений, я могу дать ответ из гугла, я с этим НЕ РАБОТАЛ

При выборе СУБД для кластера из 1000 машин основным критерием является Теорема CAP. Она гласит, что в распределенной системе невозможно одновременно обеспечить более двух из трех свойств:

C (Consistency) — Согласованность: Данные на всех узлах одинаковы в любой момент времени.

A (Availability) — Доступность: Каждый запрос получает успешный ответ (даже если данные не самые свежие).

P (Partition Tolerance) — Устойчивость к разделению: Система продолжает работать, даже если связь между узлами прервана.
